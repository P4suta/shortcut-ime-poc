# リランキング・ロードマップ Stage 0: 変換精度ベースライン（2026-06-07）

[リランキング・ロードマップ](../../../.claude/plans/wfst-mozc-mozc-embedding-bert-ime-encode-adaptive-cloud.md) の Stage 0。
「試行錯誤」を反証可能にするための測定ハーネスを据え、現行 1-best 変換器の**文単位**変換精度ベースラインを採取した。
（旧 `step0-measurement.md` は索引設計のための衝突分布測定。本書は変換精度の測定で別物。）

## ハーネス
- 指標ロジック・ハーネスは `src/ShortcutIme.Evaluation`（Core 非依存の純粋関数。`ConversionMetrics`／`EvaluationHarness`／`EvalDataset`）。各 Stage の比較で使い回す。
- 実行は `tools/ShortcutIme.Eval`。読み付きテストセット（`data/eval/seed.tsv`、TSV: 表層文＜TAB＞読み）を `ConsonantEncoder`／`RomajiEncoder` で合成入力に変換し、`PhraseConverter` に通して集計。
- 変換器設定は**製品（MainPageViewModel）と同一**: `PhraseConverter(seg=0, vowelSkip=500)`。
- 指標: 文一致（top-1）／文字精度（1−正規化レーベンシュタイン）／MRR／変換成功率。MRR・top-k は **Stage 1 の n-best 化までは top-1 と一致**（現行は 1-best のため）。
- 再現: `dotnet run -c Release --project tools/ShortcutIme.Eval -- data/dictionary_oss data/eval/seed.tsv`
- テストセットは長文寄り・活用込みの手作り 32 文（暫定）。大規模化（コーパス＋形態素解析で読み付与）は Stage 0c で Python 前処理に置換予定。

## ⚠ 子音入力の生成バグを修正（測定の前提）
初版の `ConsonantEncoder` は「母音モーラは**語頭のみ**残し以降は省略」という規則で、**文中の音節母音（家=いえ の「い」、お忙しい の「お」等）まで落としていた**。押せる子音が無いモーラはユーザが母音を打つしかないため、これは「目標に対応しない入力＝出題ミス」だった（例: `週末は家で…` が `symthdykrsms` と生成され「いえ」が消失）。
修正は2段階:
1. **子音を持たない母音モーラ（あいうえお）は母音字を残す**（押せる子音が無いため）。子音を持つモーラは頭子音のみ。
2. 長音の脱落は**確信度の高い「う」長音（お段／う段＋う＝おう・うう）と長音記号「ー」のみ**に限定。`おお・えい・同母音` などは助詞・語境界をまたいだ別音節（`を＋お`＝送って／おかけ、`て＋い`＝〜ていただく 等）を誤脱落させないため**保守的に残す**（母音を多めに残すぶんには「打てる入力」だが、落とすと打てない＝出題ミスになる）。

例: きょう→ky、きょういく→kyik、あい→ai、しゅうまつ→symt、せんせい→snsi、週末…→`symthiedykrsms`。これで子音入力が実打鍵を忠実に表すようになった。

## ベースライン（現行 1-best 変換器・子音生成バグ修正後）
| 入力モード | 文一致(top-1) | 文字精度(平均) | MRR | 変換成功 |
|-----------|--------------|---------------|-----|---------|
| 子音のみ | **15/32 (47%)** | 78.4% | 0.469 | 32/32 |
| フルローマ字 | **23/32 (72%)** | 93.0% | 0.719 | 32/32 |

（参考：生成バグ修正の過程：初版 13/32(70.9%) → 音節母音を残す 14/32(76.7%) → 長音規則を保守化 15/32(78.4%)。`本日はお忙しい中…`・`予定を確認します` 等の偽 X が解消。フルは RomajiEncoder で全母音を打つため不変。）

## 主要な発見（ロードマップへの含意）
1. **子音 44% ⇔ フル 72% の差がリランカーの狙い目**。曖昧性が増す子音入力ほど順位付けの価値が高い、という仮説を数値で裏付けた。
2. **子音の誤変換の多くは「活用語尾／文脈」の取り違え**で、文字精度 78–92% と惜しい（＝正解が n-best 近傍にいる可能性が高い）。例: `申し訳ございません→申し訳ございますね`、`お返事お待ちしております→…お待ち性ております`、`大変お世話になりました→…お世話ななりました`、`了解しました→領域しました`。→ **Stage 1（n-best 化＋`gold∈n-best` 率の計測）→ リランカー**の筋を支持。
3. **フル入力でも 72% にとどまる**（長文では「打てば素直に出る」が完全ではない）。内訳は注釈ミスではなく実誤り:
   - コスト由来の誤り: `担当者に→担当者2`（読み に に数字エントリ「2」が低コスト）、`打ち合わせの時間→打ち合わせのいい時間`（余分な文節）。→ 連接コスト調整／リランカーの対象。
   - **ローマ字方式が促音「っ」・長音「ー」を脱落させる構造的ロス**: `買った→かた`(`kata`)、`送って→おくて`(`okute`)。このため `買った/勝った/刈った` 等が同一表記へ縮約され、フル入力でも曖昧。→ Stage 2 のモデル入力設計（読み・活用型を明示的に渡す）と Stage 3（高頻度活用形の直接語彙化）に直結する論点。
4. 現行は 1-best のため MRR・top-k はプレースホルダ。Stage 1 で n-best を出して初めて「正解が何位にいるか（リランカーの上限）」が測れる。

## 多方式入力（訓令式／ヘボン式／混在）
実ユーザは方式をごちゃまぜに打つ（し=shi だが つ=tu 等）。逆変換器＝テスト生成器の一部として、これを正確に扱う:
- `RomajiScheme`（訓令/ヘボン）を `KanaRomanization`・`RomajiEncoder`・`ConsonantEncoder` に導入。
- `RomajiVariants.ExpandReading` が**モーラ単位の異形を直積**で展開（促音重ね込み・上限cap）。トライはこの全異形を索引するため、**混在打鍵でも引ける**。生成と索引が同じ表から来るので「生成入力は必ず照合可能」の不変条件が保たれる（入力正規化は不可——子音だけだと `sh` が し/す+は で曖昧）。
- 全辞書 129万×2方式＝258万検査の不変条件監査で違反は 2 件（特殊 `ゔ` のみ）。
- 多方式ベースライン（seed 32文・1-best）: 訓令式 子音47%/フル72%、ヘボン式 子音47%/フル75%。

## メモリ最適化（辞書索引の事前処理）
多方式索引は素朴なポインタ木だと 1.24GB。実測で `Dictionary<char,Node>` 417万個が約半分を占めると判明 → 圧縮:
- **Tier1**: `RomajiTrie` を **CSR フラット配列**（`childStart`/`edgeLabel(byte)`/`childTarget` ＋ 候補は `candStart`/`candId`）に書き換え、per-node 辞書を廃止。候補は record 値等価で **dedup**、表層/読みは **intern**。→ **1,237MB → ~130MB（9.5倍減）**。セマンティクスは Core/Eval 全テスト＋Eval 数値一致で維持を確認。
- **Tier2**: バイナリ**直列化**（`Save`/`Load`、int 配列は `MemoryMarshal` で一括、文字列はプール化）。→ blob **121MB**、**ロード 427ms（再構築 17s の約40倍速）**。
- 残し（必要時のみ）: パス圧縮、mmap ゼロコピー、double-array/LOUDS 簡潔木（.NET に成熟ライブラリ無く ROI 低、現状フットプリントで不要）。
